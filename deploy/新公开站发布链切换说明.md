# 新公开站发布链切换说明

## 背景

此前正式发布链仍然依赖旧 `frontend`：

- CI 先构建旧 `frontend`
- Go 后端通过 `embed` 把前端静态产物打进 `subiohub` 二进制
- Docker/GoReleaser 发布的正式镜像默认承载旧公开页

这会导致后续在 `next-web` 上完成的改动无法进入正式镜像，包括但不限于：

- 公开页路由级国际化
- SEO 元数据与 `html lang`
- 多语言 sitemap / robots
- 资讯栏目与详情页优化

本次改造的目标，就是把正式发布链彻底切到 `next-web`，确保公开页能力与生产镜像保持一致。

## 新发布架构

正式 Docker 部署现在统一为三层结构：

```text
Browser
  -> Caddy (`web`)
    -> Next.js (`next-web`)
      -> Go API (`subiohub`)
```

职责划分如下：

- `web`：统一公网入口，负责反向代理
- `next-web`：承载公开页、SSR、SEO、国际化、资讯页
- `subiohub`：只负责 API、后台能力、业务逻辑

## 本次修改范围

### 1. 后端镜像改为纯 API 模式

移除正式构建中对旧 `frontend` 的依赖：

- 根目录 [Dockerfile](file:///d:/guoziji/apirouter/subiohub/Dockerfile)
- 部署用 [deploy/Dockerfile](file:///d:/guoziji/apirouter/subiohub/deploy/Dockerfile)

- GoReleaser 简化配置 [\.goreleaser.simple.yaml](file:///d:/guoziji/apirouter/subiohub/.goreleaser.simple.yaml)

关键变化：

- 不再构建旧 `frontend`
- 不再向 Go build 传 `-tags=embed`
- `subiohub` 镜像只发布后端服务

### 2. 新增 `next-web` 独立镜像构建

新增 [next-web/Dockerfile](file:///d:/guoziji/apirouter/subiohub/next-web/Dockerfile)，采用 Next.js standalone 产物发布：

- 构建阶段执行 `npm ci` 和 `npm run build`
- 运行阶段仅拷贝 `.next/standalone`、`.next/static`、`public`
- 容器默认监听 `3000`

这意味着正式前端镜像和后端镜像已经完全解耦。

### 3. CI 发布链切换到 `next-web`

修改 [release.yml](file:///d:/guoziji/apirouter/subiohub/.github/workflows/release.yml)：

- 用 `npm ci` / `npm run build` 校验 `next-web`
- 新增 `subiohub-next-web` 镜像构建与推送
- 后端 GoReleaser 继续负责 `subiohub` 镜像与二进制发布
- 简化版发布同样不再依赖旧 `frontend`

修改 [security-scan.yml](file:///d:/guoziji/apirouter/subiohub/.github/workflows/security-scan.yml)：

- 安全扫描目标由旧 `frontend` 切换为 `next-web`
- 包管理器从 `pnpm` 切到 `npm`

## 新增和调整的环境变量

已同步到 [deploy/.env.example](file:///d:/guoziji/apirouter/subiohub/deploy/.env.example)：

| 变量名 | 用途 |
|---|---|
| `NEXT_PUBLIC_SITE_URL` | 公开站点 canonical URL，供 `next-web` SEO、sitemap、metadata 使用 |
| `NEXT_SERVER_API_ORIGIN` | `next-web` 容器内服务端请求后端时使用的内部地址 |
| `NEXT_PUBLIC_API_URL` | 浏览器侧 API 基地址，可留空走同源 |
| `CADDY_SITE_ADDRESS` | Caddy 监听地址 |
| `SERVER_FRONTEND_URL` | 后端生成外部链接时使用的公开前端地址 |

注意：

- Docker 部署时，`SERVER_FRONTEND_URL` 应与 `NEXT_PUBLIC_SITE_URL` 保持一致
- 二进制或自定义配置部署时，`server.frontend_url` 也应指向 `next-web` 的公网地址

## 升级注意事项

旧部署升级到新部署模型时，请重点确认：

1. 已拉取到新的 compose 文件和 `Caddyfile`
2. 已设置 `NEXT_PUBLIC_SITE_URL`
3. 后端 `SERVER_FRONTEND_URL` 或 `server.frontend_url` 已改为公开前端地址
4. 已允许新增 `web`、`next-web` 两个服务参与启动
5. 若使用本地目录部署，请确保迁移 `caddy_data`、`caddy_config`

## 结论

本次改造完成后，正式发布链已从旧 `frontend` 迁移到 `next-web`。  
后续所有面向公开站点的改造，应默认以 `next-web` 为正式发布载体。
