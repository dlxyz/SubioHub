# 公开站发布链说明

## 当前架构

正式 Docker 部署统一采用三层结构：

```text
Browser
  -> Caddy (`web`)
    -> Next.js (`next-web`)
      -> Go API (`subiohub`)
```

职责划分如下：

- `web`：统一公网入口，负责反向代理与站点入口
- `next-web`：承载公开页、SSR、SEO、国际化、资讯页与 Web 端界面
- `subiohub`：负责 API、管理后台接口与核心业务逻辑

默认服务关系：

- `web` 对外暴露 `80`
- `next-web` 内部监听 `3000`
- `subiohub` 内部监听 `8080`
- `/api/*`、`/v1/*`、`/v1beta/*`、`/setup/*` 等路径由 Caddy 转发到 `subiohub`
- 其他页面请求默认转发到 `next-web`

## 环境变量

已同步到 [deploy/.env.example](file:///d:/guoziji/apirouter/sub2api/deploy/.env.example)：

| 变量名 | 用途 |
|---|---|
| `NEXT_PUBLIC_SITE_URL` | 公开站点 canonical URL，供 `next-web` 的 SEO、sitemap、metadata 使用 |
| `NEXT_SERVER_API_ORIGIN` | `next-web` 容器内服务端请求后端时使用的内部地址 |
| `NEXT_PUBLIC_API_URL` | 浏览器侧 API 基地址，可留空走同源 |
| `CADDY_SITE_ADDRESS` | Caddy 监听地址 |
| `SERVER_FRONTEND_URL` | 后端生成外部链接时使用的站点地址 |

注意：

- Docker 部署时，`SERVER_FRONTEND_URL` 应与 `NEXT_PUBLIC_SITE_URL` 保持一致
- 二进制或自定义配置部署时，`server.frontend_url` 也应指向站点公网地址

对应说明已补充到 [deploy/config.example.yaml](file:///d:/guoziji/apirouter/sub2api/deploy/config.example.yaml)

## 部署脚本

## 升级注意事项

升级部署时请确认：

1. 已拉取最新 compose 文件和 `Caddyfile`
2. 已设置 `NEXT_PUBLIC_SITE_URL`
3. 后端 `SERVER_FRONTEND_URL` 或 `server.frontend_url` 已改为站点公网地址
4. 已允许 `web`、`next-web`、`subiohub` 三个服务参与启动
5. 若使用本地目录部署，已准备 `caddy_data`、`caddy_config`

## 结论

后续所有面向公开站点、用户中心和管理后台的 Web 交付，应默认以 `next-web` 作为唯一入口。
