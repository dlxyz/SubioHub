# subiohub 分销与佣金系统核心底层实现记录

本文档记录了 `subiohub` 项目分销与佣金系统底层（数据库层、仓库层、服务层、控制器层、回调层）的核心实现过程，以供后续二次开发与维护参考。

## 1. 架构模式：冻结期模式 (Pending Period)
为了防范“刷单-退款”漏洞，本次佣金系统采用了**冻结期模式**：
1. **产生佣金**：下级充值成功后，系统即刻为邀请人生成一条佣金流水，但状态为 `pending`（冻结中），此时邀请人能看到但无法提现。
2. **结算佣金**：满足冻结期（如 7 天后无退款），系统将状态更新为 `settled`，并把该金额加到邀请人的 `commission_balance`（可提现/划转余额）中。
3. **退款扣回**：如果在冻结期内发生退款，直接将流水状态改为 `cancelled`；如果已经结算后退款，系统从邀请人余额中扣除对应金额，并将流水状态改为 `reversed`。

## 2. 数据库设计 (Phase 1)
在 `ent/schema` 目录下对模型进行了修改与新增，并执行了 `go generate ./ent` 生成最新的 ORM 代码。

### 2.1 修改 `user.go`
为 `User` 实体新增了分销相关字段与边：
- **字段**：
  - `inviter_id` (int64): 邀请人的 User ID。
  - `commission_rate` (float64): 专属返佣比例 (默认 0.10 即 10%)。
  - `commission_balance` (float64): 当前可提现或划转的佣金余额。
  - `total_commission_earned` (float64): 历史累计赚取的总佣金。
- **关联 (Edges)**：
  - 添加了自关联关系：`invitees` (被邀请人列表) 与 `inviter` (邀请人)。
  - 关联到 `commission_logs`。

### 2.2 新增 `commission_log.go`
用于记录佣金流水，保障资金安全审计。
- **字段**：
  - `user_id` (int64): 受益人 (收到佣金的用户 ID)。
  - `invitee_id` (int64): 贡献佣金的下级用户 ID。
  - `order_id` (int64): 关联的充值订单 ID。
  - `amount` (float64): 本次产生的佣金金额。
  - `status` (string): 状态，包括 `pending` (冻结中)、`settled` (已结算)、`transferred` (已划转)、`withdrawn` (已提现)、`cancelled` (取消)、`reversed` (逆向扣除)。
  - `reason` (string): 来源说明。
- **关联 (Edges)**：
  - `user` (受益人)、`invitee` (贡献者)、`payment_order` (关联订单)。

### 2.3 修改 `payment_order.go`
- 新增 `commission_logs` 的边，关联到 `CommissionLog`。

## 3. 数据库访问层 - Repository (Phase 2)

### 3.1 `commission_repo.go`
实现了针对佣金流水的数据库操作接口 `CommissionRepo`：
- `Create`：创建佣金流水。
- `UpdateStatus`：更新流水状态（如从 pending 变为 settled）。
- `ListWithFilters`：带过滤条件与分页的流水查询。
- `GetPendingAmount`：利用聚合函数 `ent.Sum` 获取某用户当前所有的待结算（`pending`）佣金总额。

### 3.2 修改 `user_repo.go`
新增了安全更新用户佣金余额的方法：
- `UpdateCommissionBalance`：利用数据库层面的原子加法（`AddCommissionBalance`），避免并发下的数据脏读覆盖。如果增加的金额大于 0，还会同步累加 `total_commission_earned`。

## 4. 核心业务逻辑层 - Service (Phase 3)

### 4.1 `affiliate_service.go`
封装了分销的生命周期管理：
- `BindInviter`：绑定上下级关系。
- `RecordPendingCommission`：记录冻结佣金（充值成功时调用）。
- `SettleCommission`：结算佣金（需要事务支持，更新流水状态并给邀请人加余额）。
- `ClawbackCommission`：退款逆向扣除逻辑。

## 5. 注册绑定与鉴权 - Auth Handler (Phase 4)

为了在用户注册时建立分销绑定关系，对接口进行了如下修改：
- `auth_handler.go`：在 `RegisterRequest` 结构体中新增 `AffiliateCode` 字段（前端表单/URL 参数传入的邀请人标志）。
- `auth_service.go`：在 `RegisterWithVerification` 注册流程中解析 `AffiliateCode`，如果有效，将其转化为 `InviterID` 并与新注册用户一同写入数据库。

## 6. 支付回调自动触发返佣 - Payment Fulfillment (Phase 5)

修改了 `payment_fulfillment.go` 文件，在订单支付完成（回调或主动查询成功）并履行发货（`markCompleted` 等环节后）：
- 检查订单类型（如果是余额充值或订阅购买）。
- 查找该用户的邀请人（`InviterID`）。
- 提取邀请人的专属佣金比例（`CommissionRate`）。
- 根据 `订单金额 × 佣金比例`，自动在事务/流程中通过 `entClient.CommissionLog.Create()` 创建一条状态为 `pending` 的佣金流水。

## 7. 用户端 API 接口实现 (Phase 6)

在完成底层数据链路后，我们为用户端暴露了分销相关的核心操作接口：

### 7.1 业务层扩展 (`affiliate_service.go`)
- 新增 `TransferToBalance(ctx, userID, amount)` 方法：
  - 支持将用户的可提现佣金（`commission_balance`）划转为普通账户余额（`balance`）。
  - 在数据库事务中，先校验佣金余额是否充足，随后扣除佣金并原子累加账户余额。
  - 同步创建一条状态为 `transferred` 的资金变动流水，确保账目可追溯。

### 7.2 控制器层实现 (`affiliate_handler.go`)
创建了 `AffiliateHandler` 并注入 `AffiliateService` 与 `UserService`，实现了以下三个接口：
- **`GET /api/v1/affiliate/info`**：获取分销看板数据，包括专属返佣比例、已赚取总额、当前可用佣金余额，以及通过 `GetPendingAmount` 实时计算的冻结中金额。
- **`GET /api/v1/affiliate/logs`**：获取当前用户的佣金流水记录，支持分页（Pagination）。
- **`POST /api/v1/affiliate/transfer`**：接收前端传递的金额参数（`amount`），校验通过后调用 Service 完成资金划转。

### 7.3 路由与依赖注入配置
- **路由注册 (`user.go`)**：在需要鉴权的路由组 `authenticated.Group("/affiliate")` 下注册了上述三个接口。
- **依赖注入 (`wire.go`)**：在 `internal/service/wire.go` 和 `internal/handler/wire.go` 中注册了对应的工厂方法，并重新执行 `wire ./...` 成功生成依赖关系代码。

## 8. 踩坑记录：Wire 依赖注入与测试用例修复

在将新服务注入到 Wire 依赖树时，遇到了一些典型的编译阻碍，特此记录：

### 8.1 可变参数导致的 Wire 解析失败 (`no provider found for []time.Duration`)
- **问题现象**：在重新生成 Wire 代码时，终端报错提示找不到 `[]time.Duration` 的 provider，并一路追溯到 `OAuthRefreshAPI` 等服务。
- **根本原因**：`NewOAuthRefreshAPI(...)` 构造函数使用了可变参数 `...time.Duration`。Wire 在处理可变参数时，会尝试将其视为切片类型 `[]time.Duration` 去寻找注入提供者，从而导致解析失败。
- **解决方案**：在 `internal/service/wire.go` 中新增包装器函数 `ProvideOAuthRefreshAPI`，在包装器内部手动调用 `NewOAuthRefreshAPI`，避免 Wire 直接解析可变参数构造函数。

### 8.2 Provider 签名变更导致的历史单元测试失败
- **问题现象**：在调整 `ProvideTimingWheelService` 的返回签名（从返回 `(*TimingWheelService, error)` 改为仅返回 `*TimingWheelService` 并在内部 `panic(err)`）后，引发了 `internal/service/wire_test.go` 编译与测试失败。
- **解决方案**：同步修改测试用例 `wire_test.go`，将原先对 `error` 的断言修改为对 `panic` 的捕获（利用 `defer recover()`）和断言，使测试逻辑与最新的 Provider 签名保持一致。

## 9. 管理端 API 接口实现 (Phase 7)

为方便系统管理员对整个分销与佣金体系进行全局管控，新增了一套管理端接口。

### 9.1 业务层扩展 (`affiliate_service.go`)
- **`AdminListCommissions`**：透传底层 Repository 的 `ListWithFilters`，支持管理端基于 `user_id` 和 `status` 全局检索、过滤和分页佣金流水。
- **`AdminUpdateUserCommissionRate`**：利用 `entClient.User.Update()` 原子化更新单个用户的 `commission_rate`（专属返佣比例）。
- **`SettleCommission`（复用）**：使用之前编写的强制结算方法，供管理端手动将某笔 `pending` 的流水结算为 `settled`。

### 9.2 控制器层实现 (`admin/affiliate_handler.go`)
创建了 `admin.AffiliateHandler`，提供以下接口：
- **`GET /api/v1/admin/affiliate/commissions`**：全站流水查询。
- **`POST /api/v1/admin/affiliate/commissions/:id/settle`**：单笔佣金强制手动结算。
- **`POST /api/v1/admin/affiliate/users/:id/rate`**：为特定大客户设置专属返佣比例。

### 9.3 路由与依赖注入配置
- **路由注册 (`admin.go`)**：在管理员专用的路由组 `v1.Group("/admin")` 下，新增了 `registerAffiliateRoutes` 模块。
- **依赖注入 (`handler/wire.go`)**：将 `admin.NewAffiliateHandler` 注入到 `AdminHandlers` 集合中，重新编译 `wire ./...` 并通过。

## 10. 后续开发计划 (Next Steps)

至此，后端的分销与佣金系统基础链路（包含用户端查询、划转，管理端全局审查、设置、手动结算）已经全部闭环。
后续可按需考虑加入的高级特性：

### 10.1 手动提现审批与打款流程（核心待开发）
当前系统已支持“佣金划转为内部 API 余额”，但为了支持用户将佣金转化为真实资金（法币或加密货币），后续需要设计一套完整的**提现申请与审批工作流**：
- **数据结构扩展**：
  - 新增 `withdrawal_requests` 实体表，记录提现金额、收款账号信息（支付宝账号/姓名/USDT 地址等）、当前状态（`pending_review` 待审核, `processing` 打款中, `completed` 已完成, `rejected` 已驳回）、操作人及时间戳。
- **用户端流程**：
  - 用户提交提现申请（如申请提现 100 元）。
  - 系统立即冻结（扣除）用户的可用佣金 `commission_balance`，避免重复申请或并发划转。
- **管理端流程（风控与打款）**：
  - **审核驳回**：如果管理员发现该用户存在刷单嫌疑并驳回申请，系统需将冻结的金额退回至用户的 `commission_balance`。
  - **手动打款**：审核通过后，管理员在真实世界（如通过手机支付宝/网银）向用户打款。
  - **确认结单**：打款完成后，管理员在系统后台点击“确认已打款（可选上传转账截图凭证）”，将提现申请状态标记为 `completed`，并生成最终的资金核销流水。

### 10.2 其他计划模块
1. **定时任务 (Cron Job) 模块**：支持自动结算超过 T+N 天无退款的 `pending` 佣金。
2. **统计分析报表**：基于时间维度的拉新趋势图、佣金支出图、大客户贡献榜单等可视化数据分析。
