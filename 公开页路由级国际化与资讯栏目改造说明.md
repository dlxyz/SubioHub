# 公开页路由级国际化与资讯栏目改造说明

## 本次改造目标

本次改造围绕两个目标展开：

1. 给前端公开站点增加一个「资讯」栏目，对接后台现有公告发布系统。
2. 将公开页从原来的客户端语言切换，升级为路由级国际化，提升首屏语言一致性与 SEO 友好度。

## 已完成内容

### 1. 公开资讯栏目

- 新增公开资讯接口：
  - `GET /api/v1/news`
  - `GET /api/v1/news/:id`
- 数据来源复用后台公告系统。
- 仅暴露“当前已生效”且“没有定向规则”的公告内容。
- 前端新增公开资讯列表页与详情页：
  - `/zh-CN/news`
  - `/en-US/news`
  - `/zh-CN/news/:id`
  - `/en-US/news/:id`

### 2. 公开页路由级国际化

公开页新增 `locale` 路由段，当前支持：

- `zh-CN`
- `en-US`

已接入路由级国际化的公开页包括：

- `/[locale]`
- `/[locale]/models`
- `/[locale]/news`
- `/[locale]/news/[id]`
- `/[locale]/login`
- `/[locale]/register`
- `/[locale]/forgot-password`
- `/[locale]/reset-password`

同时已为这些路由补充服务端 `metadata` 输出，包含：

- 页面标题
- 页面描述
- canonical
- `alternates.languages`
- Open Graph 基础信息

### 3. 自动重定向

新增 `middleware` 后，访问以下无语言前缀的公开路径时，会自动跳转到默认语言路径：

- `/`
- `/models`
- `/news`
- `/login`
- `/register`
- `/forgot-password`
- `/reset-password`

当前默认语言为 `zh-CN`。

## 后端改造说明

### 公开资讯接口设计

本次没有新建独立资讯表，而是复用现有公告体系，原因如下：

- 后台已经具备公告创建、编辑、发布能力。
- 公告本身已经支持生效时间控制。
- 公告已经具备定向能力，可以区分“公开资讯”和“定向通知”。

公开资讯的筛选规则：

1. 公告必须处于生效时间内。
2. 公告必须没有定向规则。

也就是说：

- 有定向规则：继续作为登录用户公告使用。
- 无定向规则：同时可以作为公开资讯展示。

- `backend/internal/handler/announcement_handler.go`
- `backend/internal/handler/dto/announcement.go`
- `backend/internal/server/routes/auth.go`

新增能力：

- `ListPublic`
- `GetPublicByID`
- `PublicAnnouncement` DTO
- `/api/v1/news` 路由注册

## 前端改造说明

### 国际化底座调整

涉及文件：

- `next-web/src/i18n/provider.tsx`
- `next-web/src/i18n/routing.ts`
- `next-web/src/i18n/use-i18n.ts`
- `next-web/src/middleware.ts`

核心思路：

1. 用 `LocaleProvider` 统一提供当前路由语言。
2. 用 `localizePath()` 统一生成公开页本地化链接。
3. 语言切换时，不只改状态，还同步替换当前 URL。
4. 中间件负责把无语言前缀的公开地址重定向到默认语言地址。
5. 中间件会透传当前路由语言到根布局，服务端首屏直接输出对应的 `html lang`。

### 资讯栏目页面

涉及文件：

- `next-web/src/lib/news-api.ts`
- `next-web/src/components/news/news-list-view.tsx`
- `next-web/src/components/news/news-detail-view.tsx`
- `next-web/src/app/news/page.tsx`
- `next-web/src/app/news/[id]/page.tsx`
- `next-web/src/app/[locale]/news/page.tsx`
- `next-web/src/app/[locale]/news/[id]/page.tsx`

页面能力：

- 列表页展示公开资讯卡片
- 详情页展示单篇资讯正文
- 支持中英文文案
- 支持明暗主题切换
- 支持在资讯页内继续切换语言并同步切换路由
- 资讯详情页服务端 metadata 会基于真实标题与正文摘要生成

### 公开页链接本地化

已将以下页面中的公开链接切换为基于当前语言生成：

- 首页
- 模型广场
- 登录页
- 注册页
- 忘记密码页
- 重置密码页

这样用户在英文路径中继续点击公开页面，不会再跳回默认语言地址。

## SEO 与静态化说明

### 当前收益

从 SEO 角度，相比原先纯客户端切语言方案，本次改造已经明显更好：

1. 不同语言拥有独立 URL。
2. 服务端首屏 `html lang` 会跟随路由语言变化。
3. 公开页内部跳转保持语言一致。
4. `generateStaticParams()` 已为公开语言段生成静态参数。
5. 已新增 `sitemap`，覆盖多语言公开页与资讯详情页。
6. 资讯详情页 metadata 已支持真实标题与正文摘要。
7. 已新增 `robots.txt`，会显式声明 `sitemap.xml` 地址。

### 仍需继续优化的点

如果后续要进一步增强收录效果，建议继续补以下能力：

1. 如果后端后续支持多语言资讯内容，可继续按语言输出不同标题和摘要。
2. 如果后端允许服务端直连数据源，可进一步把资讯页改为服务端拉取，增强首屏内容可抓取性。
3. 视需要补充 `robots.txt` 与更细粒度的 sitemap 分片。

说明：

- 全量 `npm run lint` 仍会命中项目中原有的 `admin`、`setup` 页面历史问题。
- 这些问题不是本次改造引入的。

当前这版 html lang 服务端跟随路由语言，是通过 headers() + middleware 做到的，SEO 上是正确的，搜索引擎首屏能直接看到正确语言。
但这也意味着根布局现在变成“按请求感知”的模式，不再是最纯粹的静态根布局方案；如果你后面想把“路由级国际化 + 服务端 html lang + 最大化静态化”同时做到极致，需要再做一轮更大的路由结构调整。
sitemap 和 canonical 的绝对地址依赖 NEXT_PUBLIC_SITE_URL；如果这个环境变量没配，会回退到 http://localhost:3000。
服务端抓资讯详情和列表时，优先使用 NEXT_PUBLIC_API_URL，否则走 NEXT_SERVER_API_ORIGIN，再否则回退到本地默认值，相关逻辑在 public-news-server.ts。

## 后续建议

建议下一步继续做三件事：

1. 如果要把资讯和博客完全区分，可以后续再从公告体系拆出独立内容模型。
2. 如果后端加入富文本摘要字段，可直接替代当前正文截断摘要逻辑。
3. 可按频道或年份拆分 sitemap，降低后续资讯规模变大后的单文件长度。
