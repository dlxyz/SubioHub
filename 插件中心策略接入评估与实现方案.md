# 插件中心策略接入评估与实现方案

## 1. 结论先说

当前仓库要加“插件中心”是 **可以加** 的，但要分两层理解：

- **轻量版好加**：做成“插件目录 + 配置开关 + 跳转入口 + 后端注册表”的方式，难度中等，适合先落地。
- **完整版不算特别好加**：如果要做到“插件包独立安装、独立前端页面、独立后端路由、权限隔离、升级回滚、热插拔”，就需要先抽象一层插件运行时，不建议一步到位。

换句话说：

- 现在最适合的是先做 **平台内置插件中心**。
- 先支持“官方插件/业务模块注册与启停”。
- 后面再升级成“外部插件包机制”。

## 2. 为什么说现在能加

仓库里已经有几块可复用基础：

- 后台和前台都已经有稳定的路由体系。
- 后端已有系统设置、公开设置、权限控制、管理接口分组。
- 已经有 `custom_menu_items`、`custom_endpoints` 这种“扩展入口”思路。
- 管理台是 Next.js，做“插件中心页面、插件详情页、启停开关、版本说明”都比较直接。
- 后端已有版本检查、升级、回滚能力，后续可给插件能力复用“版本管理”思路。

## 3. 当前限制

当前还不是真正插件架构，主要限制有：

- 管理台导航还是写死在 `next-web/src/app/admin/layout.tsx`。
- Next 管理后台没有现成的“动态菜单注册中心”。
- 后端虽然能配自定义菜单和自定义端点，但这更像“扩展入口”，不是“插件生命周期管理”。
- 没有统一的插件 Manifest、插件版本表、插件依赖表。
- 没有“插件安装/卸载/升级/禁用”的标准数据结构。

所以现在如果直接上“可下载插件包 + 自动装载”，开发成本会明显上升。

## 4. 推荐的第一阶段目标

第一阶段不要追求 npm/zip 形式的外部插件安装，建议先做成：

1. 新增“插件中心”后台页面。
2. 每个插件先作为仓库内一个标准模块存在。
3. 用一个统一 Manifest 描述插件信息。
4. 用数据库设置控制插件启用/停用。
5. 前端按插件注册信息显示入口。
6. 后端按插件注册信息开放能力。

这条路的优点是：

- 风险低
- 容易做权限控制
- 不会破坏现有 Next + Go 主体结构
- 后面还能平滑升级成真正插件机制

## 5. 第一阶段建议长什么样

### 5.1 插件数据模型

建议先不要单独建复杂表，先用系统设置或简单插件表。

最小可行结构：

```json
{
  "id": "news-center",
  "name": "资讯中心",
  "version": "1.0.0",
  "category": "content",
  "enabled": true,
  "builtin": true,
  "entry_admin": "/admin/plugins/news-center",
  "entry_user": "/plugins/news-center",
  "description": "资讯、多语言和 AI 翻译相关能力",
  "capabilities": ["menu", "page", "api"]
}
```

建议字段：

- `id`：插件唯一标识
- `name`：展示名称
- `version`：插件版本
- `enabled`：是否启用
- `builtin`：是否内置插件
- `entry_admin`：管理端入口
- `entry_user`：用户端入口
- `capabilities`：插件声明的能力

### 5.2 后端注册方式

建议新增一个插件注册中心，例如：

- `backend/internal/service/plugin_registry.go`
- `backend/internal/handler/admin/plugin_handler.go`

核心思路：

```go
type PluginManifest struct {
    ID          string
    Name        string
    Version     string
    Category    string
    Description string
    Builtin     bool
    Enabled     bool
    EntryAdmin  string
    EntryUser   string
    Capabilities []string
}
```

然后在服务启动时注册内置插件：

- News
- Model Plaza
- Payment
- Affiliate
- Ops

这样插件中心页可以直接列出“当前系统有哪些模块可视为插件”。

### 5.3 前端展示方式

Next 管理台建议新增：

- `next-web/src/app/admin/plugins/page.tsx`
- `next-web/src/app/admin/plugins/[id]/page.tsx`

页面显示：

- 插件名称
- 当前版本
- 是否启用
- 插件说明
- 管理端入口
- 用户端入口
- 升级方式

## 6. 版本显示怎么做

插件中心的“版本显示”建议分两层：

### 6.1 系统版本

系统版本就是当前主程序版本，已经有后端接口：

- `/api/v1/admin/system/version`
- `/api/v1/admin/system/check-updates`
- `/api/v1/admin/system/update`
- `/api/v1/admin/system/rollback`

这层适合做：

- 当前系统版本
- 最新系统版本
- 是否可升级
- 自动升级 / 手动升级说明

### 6.2 插件版本

插件版本建议先做成 Manifest 中的 `version` 字段：

- 内置插件随主程序发布版本一起升级
- 页面上显示插件当前版本
- 如果后续某插件有单独 schema/data 变更，再增加 `schema_version`

建议展示：

- `插件版本`
- `兼容系统版本`
- `最近更新时间`
- `升级方式`

## 7. 升级办法怎么做

### 7.1 第一阶段推荐方案

先不做“插件包单独升级”，而是：

- **内置插件跟随系统升级**
- 插件中心只显示“该插件随系统版本升级”
- 如果系统是 release build，允许管理员在后台做系统升级
- 如果系统是 source build，提示手动拉代码重新部署

这是当前最稳妥的方案。

### 7.2 第二阶段增强方案

如果后面真要做“插件独立升级”，建议只做元数据升级，不直接执行任意代码安装。

推荐结构：

1. 插件中心从可信源拉取插件 Manifest
2. 校验签名/校验和
3. 下载插件静态资源或配置包
4. 更新插件版本记录
5. 执行插件迁移脚本
6. 失败时回滚到上一个版本

注意：

- Go 主程序不适合像 Node 那样在运行时任意装包并热加载业务代码
- 更安全的方式是“主程序预留能力 + 插件只装配置、页面资源和策略”

## 8. 原理是什么

插件中心的本质不是“把代码丢进去自动跑”，而是三件事：

### 8.1 注册

插件先声明自己是谁、能干什么、入口在哪：

- 名称
- 版本
- 权限
- 页面入口
- API 能力

### 8.2 编排

系统根据插件注册信息，把它挂到这些位置：

- 管理台菜单
- 用户端菜单
- 公开页入口
- 后端 API 路由
- 设置中心

### 8.3 治理

系统还要知道如何控制插件：

- 启用
- 停用
- 升级
- 回滚
- 权限边界
- 日志与审计

所以“插件中心”的核心原理是：

**Manifest + Registry + Runtime Settings + Permission + Lifecycle**

## 9. 结合当前仓库，最合适的接入路线

### 阶段 A：先做插件中心目录页

目标：

- 看得见插件
- 能启停
- 能跳转
- 能显示版本和升级说明

实现：

- 新增插件注册表
- 新增后台插件中心页
- 新增系统设置里的插件启用状态
- 后台导航加“插件中心”

### 阶段 B：做菜单动态注册

目标：

- 插件启用后自动出现在管理台导航
- 插件停用后自动隐藏

实现：

- 把 `admin/layout.tsx` 的静态菜单改成“基础菜单 + 插件菜单”
- 插件菜单来自插件中心接口

### 阶段 C：做插件能力隔离

目标：

- 每个插件声明需要哪些权限
- API 访问可单独审计

实现：

- 插件 Manifest 增加 `permissions`
- 后端中间件增加插件作用域

### 阶段 D：再考虑独立升级

只有前 3 个阶段稳定后，才建议考虑：

- 远程插件仓库
- 插件包下载
- 插件签名校验
- 插件迁移与回滚

## 10. 是否好加

最终判断：

- **做“插件中心页面 + 插件注册表 + 启停 + 版本显示 + 升级说明”是好加的**
- **做“真正外部插件安装系统”当前不算好加**

所以建议你下一步就按这个方向推进：

1. 先把插件中心当成“平台模块中心”
2. 先把 News、Model Plaza、Payment、Ops 这些能力纳入插件中心展示
3. 先实现版本显示、启停、入口和升级说明
4. 后续再抽象成真正插件机制

## 11. 我建议的下一步开发顺序

建议按下面顺序做：

1. 新增后台 `插件中心` 页面
2. 新增后端 `插件注册表 + 列表接口`
3. 先把内置模块映射成插件数据
4. 页面显示插件版本、状态、入口、升级方式
5. 导航改造成支持动态插件菜单
6. 再决定是否做插件单独升级

## 12. 一句话判断

**现在最适合加的是“内置模块型插件中心”，不是“运行时代码热插拔型插件系统”。**
