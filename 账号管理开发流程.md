# 账号管理开发流程

## 1. 目标

账号管理是当前管理后台里最复杂的模块之一，复杂点主要来自三层差异同时存在：

- `平台差异`：Anthropic / OpenAI / Gemini / Antigravity
- `账号类型差异`：OAuth 类、API Key 类、Bedrock 类、上游代理类
- `授权方式差异`：OAuth、Setup Token、API Key、SigV4 等

因此这一块不适合一次性平铺实现，必须按既有后台的分步向导和字段布局逐层对齐。

本流程文档用于约束 `next-web` 的账号管理迁移顺序，避免后续开发出现：

- 字段顺序与既有后台不一致
- 平台切换时字段错乱
- 某些账号类型漏字段
- 先实现了提交，后面又因为表单结构不一致返工

配套使用说明文档：

- `账号管理使用说明_五大平台认证与配置.md`

## 2. 总体迁移原则

### 2.1 UI 原则

- 保持既有后台的 `顶部筛选 + 列表 + 添加账号分步弹窗` 结构
- 添加账号使用 `Step 1 / Step 2` 分步向导，不改成单屏超长表单
- 平台卡片、账号类型卡片、授权方式切换区、配额控制区等布局尽量对齐原始位置

### 2.2 数据原则

- 不做“统一成一个简单表单”的偷懒收敛
- 保留后端 `credentials` 和 `extra` 的原始结构语义
- 平台差异逻辑前置到表单构建阶段，不把所有平台字段混在一起提交

### 2.3 实现原则

- 先搭骨架，再补字段，再补高级控制，再补批量能力
- 每个阶段都保证页面“可运行、可验证、可回退”
- 复杂能力必须先写到文档，再进入页面实现

## 3. 既有后台结构拆解

既有后台的账号管理可以拆成 4 个层级：

### 3.1 列表层

包含：

- 搜索账号
- 平台筛选
- 类型筛选
- 状态筛选
- Privacy 模式筛选
- 分组筛选
- 自动刷新
- 错误透传规则
- TLS 指纹模板
- 列设置
- 从 CRS 同步
- 导入 / 导出
- 添加账号

### 3.2 添加账号向导第一步

主要负责：

- 账号名称
- 备注
- 平台选择
- 账号类型选择
- 授权方式选择
- 平台相关的基础控制项
- 配额与调度基础字段
- 分组绑定

### 3.3 添加账号向导第二步

主要负责：

- OAuth 跳转
- Token 输入
- Session / Refresh Token 验证
- 平台授权回调后创建账号

### 3.4 编辑账号弹窗

主要负责：

- 基础信息编辑
- 凭证更新
- 模型限制 / 模型映射
- 配额类设置
- 高级能力开关

## 4. 平台与账号类型矩阵

### 4.1 Anthropic

支持的主要账号类型：

- `Claude Code`
  - OAuth
  - Setup Token
- `Claude Console`
  - API Key
- `AWS Bedrock`
  - SigV4 / API Key

需要重点对齐的字段：

- 模型白名单 / 模型映射
- 5h 窗口费用控制
- 会话数量控制
- RPM 限制
- 用户消息限速
- TLS 指纹伪装
- 会话 ID 伪装
- 缓存 TTL 强制替换
- 自定义转发地址
- 临时不可调度规则
- 拦截预热请求

### 4.2 OpenAI

支持的主要账号类型：

- OAuth 类账号
- API Key

需要重点对齐的字段：

- passthrough 模式
- OpenAI WebSocket v2 模式
- Codex CLI only
- 模型白名单 / 模型映射
- 自定义错误码

### 4.3 Gemini

支持的主要账号类型：

- Google One OAuth
- Code Assist OAuth
- AI Studio OAuth
- AI Studio API Key

需要重点对齐的字段：

- OAuth 类型切换
- tier 选择
- 模型映射 / 模型白名单

### 4.4 Antigravity

支持的主要账号类型：

- OAuth
- Upstream 中继

需要重点对齐的字段：

- mixed scheduling
- allow overages
- upstream base url / api key
- 默认映射
- 混合渠道风险校验

## 5. 通用字段分组

不论平台差异，账号表单里都存在一批通用字段，应优先抽成统一区块：

### 5.1 基础信息

- 账号名称
- 备注
- 平台
- 账号类型
- 授权方式

### 5.2 调度与资源

- 代理
- 并发数
- 负载因子
- 优先级
- 账号计费倍率
- 过期时间
- 过期自动暂停调度

### 5.3 分组

- 分组多选
- 默认分组自动绑定逻辑
- 混合渠道风险检查

### 5.4 状态与运行信息

- 状态
- 可调度
- 最近使用
- 错误信息
- 速率限制状态

## 6. 推荐分阶段实现顺序

为了保证迭代可控，账号管理按下面顺序实现。

### Phase 1：基础框架

目标：

- 把当前简化版账号页重构成与既有后台一致的大框架
- 只做列表区和“添加账号 Step 1”骨架

内容：

- 顶部筛选栏
- 列表列结构对齐
- 添加账号弹窗 Step 1
- 平台卡片
- 账号类型卡片
- 授权方式切换
- 基础通用字段区

### Phase 2：Anthropic 主路径

目标：

- 先打通最复杂、最核心的 Anthropic 路径

内容：

- Claude Code OAuth
- Claude Code Setup Token
- Claude Console API Key
- AWS Bedrock
- 5h 窗口费用控制
- RPM 限制
- TLS 指纹 / Session ID / TTL / 自定义转发

### Phase 3：OpenAI / Gemini / Antigravity

目标：

- 分平台补齐特有字段和授权流程

内容：

- OpenAI passthrough / ws mode / codex cli
- Gemini OAuth 类型切换与 tier
- Antigravity upstream / mixed scheduling / allow overages

### Phase 4：编辑账号与高级能力

目标：

- 补齐编辑账号弹窗
- 支持对已有账号进行同结构编辑

内容：

- 凭证更新
- 模型白名单 / 模型映射编辑
- 高级控制项回填与保存

### Phase 5：批量能力与辅助工具

目标：

- 补齐既有后台的运营型工具按钮

内容：

- 自动刷新
- 列设置
- 从 CRS 同步
- 导入 / 导出
- 错误透传规则
- TLS 指纹模板入口

## 7. `next-web` 当前实现状态

当前 `next-web` 账号管理已完成 `Phase 1`，并已进入 `Phase 2` 的第一步，现阶段已具备：

- 统一顶部筛选栏骨架
- 列表列结构骨架
- 自动刷新入口

  - 5h 窗口费用控制
  - 会话数量控制

- 代理列表真实下拉数据（`/admin/proxies/all`）

- `Gemini OAuth` 的账号等级、AI Studio 可用性提示、公共区块联动
- `Gemini OAuth` 第二步授权链接生成 / 授权码兑换 / 当前页配置联动创建

- `Gemini API Key` 的 `使用帮助 / 获取 API Key / base_url / api_key / aistudio tier`
- `Gemini API Key` 的模型限制 / 自定义模型 / 池模式 / 自定义错误码 / 基础配额限制
- `Antigravity OAuth` 的当前页模型映射区、默认映射加载、快捷预设映射按钮

- `Antigravity` 的 `base_url / api_key / mixed_scheduling / allow_overages`

- 其他平台的专属凭证表单

当前还没补的是 `Claude Console` 更完整配额配置、`Bedrock` 的更完整配额与通知配置、其他平台专属表单、编辑账号弹窗，以及导入导出 / 同步 / 列设置。

## 8. 接下来每轮开发的建议节奏

账号管理不建议一口气全部实现，建议每轮只聚焦一个清晰子目标。

推荐节奏：

1. 已完成 `Phase 1`
2. 已完成 `Anthropic` 的 `Claude Code` 首批授权与控制项
3. 已接入 `Claude Console` API Key 表单和模型限制 / 池模式 / 自定义错误码
4. 已接入 `Bedrock` 当前页表单与直接创建
5. 已接入 `OpenAI API Key` 当前页表单与直接创建
6. 已接入 `OpenAI OAuth` 授权流与专属控制项
7. 已接入 `Gemini OAuth` 授权流与专属控制项
8. 已接入 `Gemini API Key` 当前页表单与直接创建
9. 已接入 `Antigravity` 的 `OAuth / Upstream`、默认映射、快捷映射与专属开关
10. 下一步回头补齐 `Claude Console / Bedrock` 更完整配额配置，并继续推进编辑账号弹窗

## 9. 本文档的使用方式

后续每次继续开发账号管理时，都应先做两件事：

1. 先在本文档中确认本轮属于哪个 `Phase`
2. 只在该 `Phase` 范围内新增字段和交互

这样能保证：

- 不会把大功能做散
- 不会把不同平台字段混在一起
- 不会出现 UI 已经换了，但数据结构还没对齐的问题
