# subiohub 二级代理分销与推广提佣系统设计文档

## 1. 需求概述
本计划旨在对开源的 `subiohub` 系统进行二次开发，增加一套闭源的**多级代理销售与推广提佣功能**。
核心流程：
1. **推广与绑定**：用户可生成专属邀请链接或邀请码，新用户通过该链接注册即与推广人建立“上下级绑定关系”。
2. **消费与返佣**：当下级用户进行充值或消费时，系统根据设定的比例（全局比例或单独配置的代理比例）自动计算佣金并返还给上级代理。
3. **佣金与提现**：代理可查看自己的推广明细和佣金流水，当佣金达到设定阈值时，可申请提现。管理员审核后打款。

## 2. 数据库扩展设计 (Go/ent)

### 2.1 用户表扩展 (`users`)
在现有的用户表中新增以下字段：
- `inviter_id` (int/uuid): 邀请人ID（上级代理），建立树状关系。
- `invite_code` (string): 用户的专属邀请码（唯一，生成账号时自动生成）。
- `commission_rate` (float): 该用户的专属返佣比例（如 0.2 代表 20%）。如果不设置，则使用系统全局默认比例。
- `commission_balance` (float): 当前可提现佣金余额。

### 2.2 新增佣金流水表 (`commission_logs`)
用于记录每一笔返佣的来源和明细：
- `id`: 流水ID
- `user_id`: 获得佣金的代理ID
- `invitee_id`: 产生佣金的下级用户ID；当流水来源是“佣金划转到余额”等非邀请场景时允许为空
- `amount`: 佣金金额
- `order_id`: 关联的充值或消费订单ID
- `status`: 状态（如 `pending`、`settled`、`transferred`、`withdrawn`）
- `reason`: 描述（例如“推广奖励：下级充值 $100”）
- `created_at`: 创建时间

### 2.3 新增提现申请表 (`withdrawals`)
用于代理申请佣金提现及管理员审核：
- `id`: 提现申请ID
- `user_id`: 申请提现的用户ID
- `amount`: 申请提现的金额
- `status`: 状态 (`pending` 待审核, `approved` 已通过/已打款, `rejected` 已拒绝)
- `payment_method`: 收款方式（如支付宝、USDT、银行卡）
- `payment_account`: 收款账号详情
- `reject_reason`: 拒绝理由（可选）
- `created_at`: 申请时间
- `processed_at`: 处理时间

## 3. 后端接口设计 (Go/Gin)

### 3.1 客户端接口 (User API)
- `GET /api/user/affiliate/info`: 获取当前用户的邀请码、邀请链接、佣金余额、总收益、下级人数等统计信息。
- `GET /api/user/affiliate/downlines`: 获取直接下级用户列表（可脱敏显示用户名、注册时间）。
- `GET /api/user/affiliate/records`: 获取佣金收益明细列表。
- `POST /api/user/affiliate/withdraw`: 提交提现申请（传入提现金额、收款方式及账号）。
- `GET /api/user/affiliate/withdrawals`: 获取提现申请记录及状态。

### 3.2 管理端接口 (Admin API)
- `GET /api/admin/affiliate/users`: 查看所有用户的分销数据（下级人数、累计收益等），并支持修改指定用户的 `commission_rate`。
- `GET /api/admin/affiliate/withdrawals`: 获取所有用户的提现申请列表。
- `POST /api/admin/affiliate/withdrawals/:id/audit`: 审核提现申请（通过/拒绝）。
- `POST /api/admin/config/affiliate`: 设置全局默认返佣比例、最低提现金额限制等全局参数。

### 3.3 核心业务逻辑钩子
- **注册拦截**：在用户注册接口（Register）中，解析请求中的 `affiliate_code` 参数，将其映射到邀请人的 `invite_code`，验证有效性后写入 `inviter_id`。
- **充值/扣费钩子**：在处理充值成功回调或订单支付成功的逻辑后，触发异步计算：
  1. 查找支付用户的 `inviter_id`。
  2. 若存在上级，根据上级的 `commission_rate`（或全局比例）计算佣金。
  3. 增加上级的 `commission_balance`，并插入一条 `commission_logs` 记录。

## 4. 前端页面设计 (Vue3)

### 4.1 用户端 (User Dashboard)
新增 **“推广中心” (Affiliate Center)** 菜单：
- **总览面板**：展示专属邀请链接（带一键复制功能）、可提现佣金、累计收益、下级总人数。
- **推广明细页**：Tab切换，分别展示“我的下级列表”和“收益流水”。
- **提现管理页**：提供提现申请表单，展示历史提现记录及状态进度条。

### 4.2 管理端 (Admin Dashboard)
新增 **“分销管理” (Affiliate Management)** 菜单：
- **全局设置**：配置默认返佣比例、提现门槛、提现说明等。
- **代理列表**：查看系统中所有有下级的用户，支持为其单独设置特殊返佣比例。
- **提现审核**：以列表形式展示待处理的提现申请，支持管理员点击“同意（标记为已打款）”或“驳回（填写理由，佣金退回余额）”。

## 4.3 当前 Next.js 迁移衔接情况

虽然本方案最初按 Vue3 页面设计，但当前工程已经进入 `next-web` 迁移阶段，分销相关功能需同步按新前端结构规划：
1. **用户端挂载位置**：
   - 分销入口已在个人用户中心 `/dashboard` 中保留，并作为用户中心主导航与左侧菜单的一部分。
   - 当前用户端 `Affiliate` 页面已具备邀请链接复制、佣金概览、佣金划转等基础展示能力，并已接入双语切换框架。
   - 当前邀请链接实际使用 `invite_code` 生成 `/register?affiliate_code=<invite_code>`，不是直接拼接 `user_id`。
2. **管理端挂载位置**：
   - `分销管理` 已纳入 `next-web/src/app/admin/layout.tsx` 的真实导航，不再只是设计稿中的预留菜单。
   - 当前后台已完成首个真实版本，支持 `佣金流水筛选`、`手动结算`、`用户返佣比例设置`。
   - `系统设置 -> 分销结算` 现已提供 `自动结算` 与 `手动打款后结算` 两个开关，其中后者关闭后，分销管理页将只保留查看能力，不允许继续执行手动结算。
   - 自动结算服务已在后端实现，但是否运行依赖系统配置；因此管理端手动结算仍需保留作为兜底入口。
3. **多语言要求**：
   - 后续无论继续补 Vue 还是 Next 页面，分销相关文案都应同步接入当前 `next-web` 的 i18n 体系，避免用户端与管理端一边支持双语、一边仍为写死中文。
4. **开发建议**：
   - 方案层面仍以“后端先稳定接口、前端再对齐页面”的方式推进。
   - 若未来继续完善提现审核、下级列表、收益明细等页面，应优先直接在 `next-web` 中落地，而不是继续扩展旧 Vue 端。

---

## 5. 开发步骤建议

1. **数据库层**：修改 `subiohub/backend/ent/schema` 目录，增加相关字段和新表，然后运行 `go generate ./ent` 生成代码。
2. **模型与逻辑层**：在后端编写分销计算的核心服务（Service）。
3. **接口层**：实现相关的 Controller 和 Router。
4. **前端对接**：优先在 `next-web` 目录下新增对应页面和管理后台模块，并与后端联调；旧 `frontend` 仅作为逻辑和交互参考，不再作为长期主前端。

### 5.1 当前阶段补充说明
1. **用户中心先对齐壳子再补业务**：
   - 当前用户中心的导航、语言切换、顶部入口已经完成，分销相关页面可以直接挂在现有 `/dashboard` 体系中继续增强。
2. **管理端按导航顺序推进**：
   - `分销管理` 已经属于真实页面模块，后续补“提现审核”“全局设置”“下级关系视图”时，应按当前后台导航和 i18n 规则继续演进。
3. **联动支付与订单模块**：
   - 因为返佣计算与充值成功、订单完成高度耦合，后续实现时应直接复用当前已迁移的 `订单管理 / 支付看板 / 系统设置` 能力，避免另起一套支付状态判断逻辑。
4. **联动运维监控模块**：
   - 当前 `next-web` 后台已补上真实 `运维监控 / 系统日志` 页面，后续若分销提现、返佣结算或佣金审计需要排查异常链路，可直接沿用现有 `admin/ops` 页面与后端运维接口做观测，而不必重新造一套后台排障入口。
