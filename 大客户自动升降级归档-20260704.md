## 3. 数据来源

自动升降级当前依赖两个核心数据：

1. 累计充值
   - 来源：用户累计正向充值汇总

2. 当月消费
   - 来源：当月 usage 实际成本聚合

说明：

- 前端大客户候选判断与后端自动同步逻辑使用的是同一套门槛定义
- 现在 `admin/users/:id/usage` 已经不是 mock 数据，而是真实聚合数据

## 4. 规则说明

### 4.1 自动升级

若开启 `自动升级`：

- 命中企业累计充值门槛，或企业月消费门槛，则目标等级为 `enterprise`
- 否则命中 VIP 累计充值门槛，或 VIP 月消费门槛，则目标等级为 `vip`

### 4.2 自动降级

若开启 `自动降级`：

- 若企业 / VIP 门槛都未命中，则目标等级回退到 `standard`

### 4.3 默认策略处理

同步过程中，策略处理规则如下：

1. 新晋升的大客户
   - 自动套用当前配置中的默认折扣和默认返点

2. 已经是大客户且仍沿用默认策略的用户
   - 若等级变化，则自动切换到新等级对应的默认策略

3. 已经被人工单独修改过折扣 / 返点的用户
   - 升级时尽量保留人工策略，不强行覆盖

4. 降为 `standard` 的用户
   - 折扣重置为 `1`
   - 返点重置为 `0`

## 5. 后端接口

### 5.1 手动触发同步

接口：

- `POST /api/v1/admin/users/key-accounts/auto-sync`

请求体：

```json
{
  "dry_run": false
}
```

字段说明：

- `dry_run=true`：仅预演，不落库
- `dry_run=false`：正式执行并落库

返回示例字段：

```json
{
  "dry_run": false,
  "auto_upgrade_enabled": true,
  "auto_downgrade_enabled": true,
  "scanned": 120,
  "eligible": 9,
  "changed": 6,
  "upgraded": 4,
  "downgraded": 2,
  "promoted_to_key_account": 3,
  "removed_from_key_account": 1,
  "applied_default_strategy": 4,
  "failed": 0,
  "items": []
}
```
