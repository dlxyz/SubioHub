# frontend 命名清理总结与资讯管理开发前清单

## 一、文档目的

这份清单用于在进入“资讯管理”开发前，先把本轮 `frontend` 命名清理工作的结果沉淀下来，避免后续开发时再次混入旧口径。

同时，这份文档也作为资讯系统开发的前置检查表，确保接下来新增的表结构、接口、后台页面和公开站页面都统一使用新的命名原则。

## 二、本轮清理结论

当前仓库已经完成以下统一：

- 项目对外口径统一为 `next-web` 是主 Web 应用。
- README、部署文档、架构说明中的旧 `frontend` 说明已大幅清理。
- 设置页、语言包、提示文案中的“前端地址”类描述已统一改为“站点地址”或 “Web”。
- 后端注释、日志、错误文案中可安全替换的旧 `frontend` 表达已改写。
- 纯展示层文档和历史迁移文档中，已尽量去除“旧 frontend / Vue 并行主线”的描述。

## 三、当前命名原则

后续新增资讯系统代码时，统一遵循以下规则：

### 1. 对外展示文案

面向用户、管理员、部署者的文案，优先使用以下表达：

- `Web`
- `站点`
- `站点地址`
- `Web 回调路径`
- `Web 页面`
- `Web 应用`

避免继续新增以下旧表达：

- `frontend`
- `前端地址`
- `前端回调路径`
- `前端页面`
- `前端系统`

### 2. 代码内部命名

新增业务代码时，优先使用：

- `siteURL`
- `webURL`
- `webCallbackPath`
- `newsWebPage`
- `publicWebRoute`

除非为了兼容既有接口，否则不要再新增：

- `frontendUrl`
- `frontendRedirectUrl`
- `frontendPage`

### 3. 兼容字段保留策略

以下内容暂时允许保留，不作为本轮强制重命名范围：

- 已上线配置键，如 `frontend_url`
- 已存在 JSON 字段，如 `frontend_url`
- 已有数据库设置键或常量名
- 依赖兼容性的 handler / service 方法名
- 测试中用于验证历史字段的断言

原因：

- 这些名称一旦整体改动，容易影响现有配置文件、接口兼容性和联调逻辑。
- 当前更重要的是先完成资讯系统功能，不建议在此时做高风险字段迁移。

## 四、已清理范围

本轮已经处理过的范围包括但不限于：

- 根目录 README 系列文档
- 架构与部署说明文档
- `deploy` 目录说明和示例配置
- `next-web` 设置页部分展示文案
- `frontend` 旧管理端中的部分语言包和注释
- 后端设置、认证、嵌入式 Web 资源相关注释与提示语

## 五、仍允许存在的 frontend 命名类型

后续如果搜索到 `frontend`，先判断是否属于下面几类：

### 1. 兼容配置项

例如：

- `frontend_url`
- `frontend_redirect_url`
- `oidc_connect_frontend_redirect_url`

处理原则：

- 可以保留字段名
- 但界面标签、帮助说明、错误提示尽量改成 `Site URL`、`Web Callback Path` 一类新口径

### 2. 旧函数名或结构体名

例如：

- `GetFrontendURL()`
- `ValidateFrontendRedirectURL()`
- `FrontendServer`
- `ServeEmbeddedFrontend()`

处理原则：

- 现阶段先不做大规模重构
- 新代码不要继续复制这类命名
- 如果未来专门做“兼容命名重构”，再单独开任务处理

### 3. 历史测试与旧目录

例如：

- 旧测试名称
- 旧 Vue 目录中的历史实现
- 为兼容接口保留的 DTO 字段

处理原则：

- 不阻塞资讯管理开发
- 后续可单独做一次低优先级清理

## 六、资讯管理开发前检查清单

进入资讯系统开发前，先逐项确认：

- [ ] 资讯系统与公告系统彻底分表，不再复用 `announcements`
- [ ] 资讯主表命名确定为 `news_posts`
- [ ] 多语言内容表命名确定为 `news_post_translations`
- [ ] 资讯公开路由继续以 `next-web` 为主线承载
- [ ] 管理后台新增资讯页面时，统一使用 `Web`、`站点` 口径
- [ ] 富文本编辑器输出格式首版统一为 HTML
- [ ] AI 翻译作为编辑器生产工具，不做公开站运行时依赖
- [ ] 公开接口按语言返回资讯正文，不再只做公告映射
- [ ] 后端返回字段中，如需兼容旧字段命名，先保留兼容，再单独规划迁移
- [ ] 新增资讯相关文档时，不再出现“旧 frontend 主线”的描述

- `Web`

- `AnnouncementNews`
- `FrontendNews`
- `NewsFrontendPage`
