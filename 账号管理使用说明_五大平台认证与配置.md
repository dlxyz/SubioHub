# 账号管理使用说明：五大平台认证、选项与模型映射

## 1. 这份文档是干什么的

这份文档是给管理员看的“实操说明书”，目标不是讲代码，而是讲：

- 账号管理里每个平台到底有什么区别
- 什么时候该选 `OAuth`，什么时候该选 `API Key`
- 每个选项大概是什么意思
- `模型白名单`、`模型映射`、`池模式`、`临时不可调度` 这些到底是干什么的
- 新建账号时，怎样用一套比较稳的思路把配置填对

目前账号管理里，实际会接触到的主要平台可以按“五大来源”来理解：

- `Anthropic`
  - `Claude Code`
  - `Claude Console`
- `AWS Bedrock`
- `OpenAI`
- `Gemini`
- `Antigravity`

其中 `Claude Code`、`Claude Console` 都属于 Anthropic 体系，但认证逻辑和适用场景差异很大，所以实际使用时要把它们当成不同类型来理解。

---

## 2. 先建立 4 个核心概念

在使用账号管理之前，先把下面 4 个概念分清楚，否则页面选项会看起来很多、很乱。

### 2.1 平台

平台就是“这个账号最终接到谁的服务”。

例如：

- `Anthropic`：Claude 官方体系
- `AWS Bedrock`：走 AWS 的模型托管
- `OpenAI`：OpenAI 官方体系
- `Gemini`：Google Gemini / AI Studio / Code Assist
- `Antigravity`：Google Cloud AI Companion 相关体系

### 2.2 账号类型

账号类型是“同一平台下，按接入来源再细分的种类”。

例如：

- Anthropic 下分：
  - `Claude Code`
  - `Claude Console`
  - `AWS Bedrock`
- OpenAI 下分：
  - `OAuth`
  - `API Key`
- Gemini 下分：
  - `OAuth`
  - `API Key`
- Antigravity 下分：
  - `OAuth`
  - `API Key`

### 2.3 认证方式

认证方式是“你拿什么东西去证明自己有权限访问这个平台”。

常见有：

- `OAuth`
  - 典型场景：浏览器登录授权后拿到 `access_token / refresh_token`
- `Setup Token`
  - 典型场景：Claude Code 类长期令牌
- `API Key`
  - 典型场景：后台直接填写密钥
- `SigV4`
  - 典型场景：AWS Access Key / Secret Key 签名

### 2.4 credentials 和 extra

页面上的字段，提交时大体会落到两类：

- `credentials`
  - 真正用于请求上游平台的凭证和运行参数
  - 例如 `api_key`、`access_token`、`refresh_token`、`base_url`、`model_mapping`
- `extra`
  - 更偏“行为开关”或“附加能力”的信息
  - 例如 `openai_passthrough`、`mixed_scheduling`、`allow_overages`

你可以简单理解为：

- `credentials` 决定“怎么连上游”
- `extra` 决定“这个账号在系统里怎么用”

## 3. 账号创建时的通用思路

不管是什么平台，都建议按这个顺序考虑：

1. 先判断你接的是哪个平台
2. 再判断你手里有什么认证材料
3. 再决定模型限制是用白名单还是映射
4. 最后再开高级选项，不要一开始全打开

推荐你第一次配置时坚持一个原则：

- `先最小可用`
- `再逐步加限制`

也就是说：

- 先把账号接通
- 再加模型限制
- 再加池模式、错误码、临时不可调度、代理等高级能力

这样最不容易把问题混在一起。

## 4. 通用字段说明

这些字段几乎所有平台都会遇到。

### 4.1 账号名称

这是后台给你自己看的名字，不会直接发给上游。

建议命名规则统一，例如：

- `claude-console-main`
- `openai-oauth-01`
- `gemini-aistudio-paid`
- `antigravity-rt-batch-01`

建议命名里带上：

- 平台
- 认证方式
- 用途
- 编号

### 4.2 备注

用于写清楚这个账号是干嘛的，例如：

- 生产环境主账号
- 测试环境
- 专门跑图像模型
- 只给某分组使用

### 4.3 代理

如果你有代理池、地区出口要求、网络隔离需求，就给账号绑定代理。

适合使用代理的情况：

- 需要固定出口 IP
- 需要通过公司代理访问海外服务
- 某个平台在特定网络环境下更稳定

不确定时可以先不配。

### 4.4 并发数

表示这个账号最多允许多少请求同时进行。

可以理解为：

- 值越大，吞吐可能越高
- 但如果上游本身限制严格，过大反而更容易报错

保守建议：

- 不熟悉时先用默认值
- 稳定后再逐渐增大

### 4.5 负载因子

这是调度器分配流量时的权重参考。

可简单理解为：

- 越大，调度器越倾向把请求打到这个账号
- 越小，越偏保守

适合提高负载因子的情况：

- 该账号额度大
- 响应快
- 稳定性高

### 4.6 优先级

优先级通常用于多个账号一起调度时，决定先选谁。

可以理解为：

- 更高优先级的账号更容易先被选中

典型用途：

- 主账号优先
- 备用账号兜底

### 4.7 账号计费倍率

这是系统内部计费/统计侧的倍率，不是上游平台的真实价格本身。

它常用于：

- 不同账号成本差异化
- 同平台不同来源的价格修正
- 对用户侧计费做运营调整

如果你暂时不做复杂计费，可以先保持默认值。

### 4.8 过期时间

到期后这个账号可以自动停止调度，适合：

- 临时测试账号
- 短期活动账号
- 已知有效期的授权账号

### 4.9 分组

分组决定“哪些用户/哪些 API Key 可以用到这批账号”。

如果你不理解分组，可以先记住：

- 账号不只是“能不能连上”
- 还涉及“给谁用”

## 5. 模型限制：白名单、模型映射、默认映射到底是什么

这是账号管理里最容易看不懂、但最重要的一块。

### 5.1 模型白名单

意思是：

- 这个账号只允许访问你勾选过的模型

适合场景：

- 只想让某账号跑少量固定模型
- 想防止高价模型被误用
- 想按账号能力做隔离

例子：

- 只允许 `gpt-4.1-mini`
- 不允许 `gpt-4.1`

### 5.2 模型映射

意思是：

- 用户请求的是模型 A
- 系统实际转发给上游的是模型 B

这不是“允许/不允许”的问题，而是“把名字改写后再发出去”。

典型用途：

- 兼容历史模型名
- 把一个大类请求统一收敛到稳定模型
- 平台之间做模型名适配

例子：

- 用户请求 `claude-*`
- 实际发送 `claude-sonnet-4-5`

再例如：

- 用户请求 `gemini-2.5*`
- 实际发送 `gemini-2.5-flash`

### 5.3 白名单和映射的区别

一句话理解：

- `白名单` = 决定“哪些能用”
- `映射` = 决定“最终用哪个”

如果你只想限制范围，用白名单。

如果你想把各种请求统一改写成固定模型，用映射。

### 5.4 默认映射

默认映射一般是系统预设的一组“官方或推荐映射规则”。

用途是：

- 帮你快速起步
- 避免第一次配时完全空白
- 让常用模型名可以直接落到合理的实际模型上

### 5.5 快捷预设映射

这是常见映射模板按钮，点击后会自动加入一条映射规则。

适合：

- 快速建立常用映射
- 不想手打复杂通配符

### 5.6 什么时候优先用映射

建议优先用映射的场景：

- `Antigravity`
- 想兼容多个外部模型名
- 想把请求统一收敛到成本更可控的模型

### 5.7 什么时候优先用白名单

建议优先用白名单的场景：

- `Claude Console`
- `OpenAI API Key`
- `Gemini API Key`
- 你想非常明确地限制可用模型集合

## 6. 通用高级选项说明

### 6.1 池模式 `pool_mode`

这个能力常见于 API Key 类账号。

可以理解为：

- 把单账号当成“一个池中的成员”
- 失败时按设定重试/切换

适合：

- 同类型账号比较多
- 想做故障切换
- 单个账号偶发失败，但整体池子可继续工作

不适合：

- 只有一个账号
- 你还在排查基础连通性

### 6.2 池模式重试次数 `pool_mode_retry_count`

表示池模式触发时，最多尝试多少次。

建议：

- 不熟悉时先用默认值
- 不要一开始就拉太高

太高可能带来：

- 响应变慢
- 错误链条更难排查

### 6.3 自定义错误码 `custom_error_codes`

这是告诉系统：

- 某些错误码你希望按“特殊失败信号”对待

典型用途：

- 让某些上游特定报错也能触发失败转移
- 适配非标准错误响应

如果你对上游错误语义不熟，建议先不要乱加。

### 6.4 临时不可调度 `temp_unschedulable`

这个非常有用。

意思是：

- 当账号命中特定错误时，先把它临时下线一段时间
- 避免调度器持续把请求打到一个已经出问题的账号上

常见场景：

- `429` 限流
- `503` 不可用
- `529` 过载

一条规则通常包含：

- 错误码
- 关键词
- 持续时间
- 描述

适合：

- 大规模账号池
- 上游偶发限流或过载

### 6.5 拦截预热请求 `intercept_warmup_requests`

适合某些平台或客户端在真正请求前，会先发“探活/预热”请求的情况。

作用是：

- 避免无意义预热影响计费或调度判断

不确定时可以先关闭。

### 6.6 自定义转发地址 `base_url`

这个字段很关键。

它不一定是“官方 API 地址”，也可能是：

- 你自己的中继服务
- 企业网关
- 自建代理层

只有在你明确知道目标地址正确时再改。

否则优先用默认地址。

## 7. 平台详解

## 7.1 Anthropic：Claude Code

### 适合什么场景

适合：

- 你接的是 Claude Code 体系
- 你拿到的是 OAuth 授权或 Setup Token
- 你需要比较细的 Anthropic 专属调优

### 可选认证

- `OAuth`
- `Setup Token`

### OAuth 和 Setup Token 怎么选

- `OAuth`
  - 更接近标准授权流
  - 适合通过浏览器授权获取凭证
- `Setup Token`
  - 更偏长期令牌思路
  - 适合你已经拿到了 Setup Token 或偏长期稳定接入

如果你没有特别理由，优先按你手头实际可获得的凭证来选，不要硬切。

### Claude Code 常见专属选项

- `5h 窗口费用控制`
  - 限制账号在 5 小时窗口内的成本使用
  - 适合控制 Anthropic 成本暴涨
- `会话数量控制`
  - 控制会话层面的占用
- `RPM 限制`
  - 限制每分钟请求数
- `用户消息限速`
  - 控制单用户请求节奏
- `TLS 指纹伪装`
  - 用于更贴近特定客户端网络特征
- `会话 ID 伪装`
  - 某些兼容链路下会用到
- `缓存 TTL 强制替换`
  - 控制缓存生存时间
- `自定义转发地址`
  - 把请求转到指定上游

### 推荐用法

第一次配置建议：

1. 先只接通 `OAuth` 或 `Setup Token`
2. 先不开 TLS、Session ID、TTL 这些高级项
3. 如果你已有账号池，再加 `临时不可调度`
4. 成本敏感时，再补 `5h 窗口费用控制`

---

## 7.2 Anthropic：Claude Console

### 适合什么场景

适合：

- 你手里是 Anthropic 官方 `API Key`
- 想用比较标准、直接的方式接 Claude

### 可选认证

- `API Key`

### 主要字段

- `Base URL`
  - 默认一般是官方地址
  - 只有用中继时才改
- `API Key`
  - 你的 Anthropic 密钥
- `模型白名单 / 模型映射`
  - 用来限制或改写模型
- `池模式`
  - 多个 Claude Console 账号一起用时比较有意义
- `自定义错误码`
  - 高级兼容用途

### 什么时候用白名单

适合你只想开放少数稳定模型时，例如：

- 只允许 `claude-sonnet-*`

### 什么时候用模型映射

适合你想把多个外部模型请求统一转到一个实际模型时，例如：

- 所有 `claude-*` 都收敛到 `claude-sonnet-4-5`

### 推荐用法

最常见、最稳的配置通常是：

1. `Base URL` 保持默认
2. 填 `API Key`
3. 先用白名单限制几种常用模型
4. 账号多时再开池模式

## 7.3 AWS Bedrock

### 适合什么场景

适合：

- 你已经在 AWS 上开通了 Bedrock
- 想通过 AWS 体系访问模型
- 你更习惯云厂商凭证体系

### 可选认证

- `SigV4`
- `API Key`

### SigV4 是什么

SigV4 是 AWS 的签名认证方式，通常需要：

- `AWS Access Key ID`
- `AWS Secret Access Key`
- 可选 `Session Token`

这类方式更适合标准 AWS 接入。

### API Key 是什么

某些情况下会提供更简单的 key 方式接入 Bedrock。

如果你已经是标准 AWS 环境，通常更常见的是 `SigV4`。

### 主要字段

- `Region`
  - 你要访问的 AWS 区域，例如 `us-east-1`
- `Force Global`
  - 强制走全局模型访问逻辑
  - 一般只有你明确知道它对你当前区域更合适时才开
- `模型白名单 / 模型映射`
- `池模式`
- `基础配额`

### 推荐用法

第一次配置建议：

1. 优先选 `SigV4`
2. `Region` 填你实际开通区域
3. 不清楚 `Force Global` 时先关闭
4. 先确保连通，再加模型限制

## 7.4 OpenAI

OpenAI 目前主要分两种：

- `OpenAI OAuth`
- `OpenAI API Key`

### 7.4.1 OpenAI API Key

#### 适合什么场景

适合：

- 你有标准 OpenAI `API Key`
- 想快速稳定接入

#### 主要字段

- `Base URL`
  - 官方地址或你的 OpenAI 兼容中继
- `API Key`
- `模型白名单 / 模型映射`
- `自动透传（仅替换认证）`
- `WS mode`
- `Codex 官方客户端限制`
- `池模式`
- `自定义错误码`

#### 自动透传 `openai_passthrough`

这个选项很重要。

意思是：

- 尽量保留原始 OpenAI 请求结构
- 系统主要只替换认证信息

适合：

- 你想最大程度保持原始 OpenAI API 行为
- 你前面已经有兼容 OpenAI 的客户端

不适合：

- 你需要系统强制改写模型映射

经验上：

- 开了“自动透传”后，通常更像“直接给 OpenAI 换个认证层”
- 如果你还要做很多模型改写，反而不一定适合开

#### WS mode

这是和 OpenAI 响应流/实时能力相关的模式开关。

可以理解为：

- 控制是否启用响应 WebSocket v2 相关兼容行为

适合：

- 你明确知道客户端依赖这类响应模式

不确定时：

- 先保持默认或关闭

#### Codex 官方客户端限制

这个选项的意思是：

- 只允许特定官方客户端家族使用

适合：

- 你希望账号只服务于 Codex 官方客户端链路

一般管理员首次使用时，不建议一开始就开。

### 7.4.2 OpenAI OAuth

#### 适合什么场景

适合：

- 你是通过 OpenAI 登录授权拿到 OAuth 凭证
- 不是单纯持有 API Key

#### 主要流程

1. 生成授权链接
2. 浏览器完成授权
3. 回填授权码
4. 系统兑换 token 后创建账号

#### 适合注意的点

- 如果只是普通接入，通常 `API Key` 更直接
- 需要官方 OAuth 场景时再选 `OAuth`

## 7.5 Gemini

Gemini 当前分为：

- `Gemini OAuth`
- `Gemini API Key`

### 7.5.1 Gemini OAuth

#### 可选类型

- `Google One`
- `GCP Code Assist`
- `AI Studio OAuth`

你可以把它理解为：

- 同样是 Google 体系授权
- 但账号来源和权限背景不同

#### 各类型大致区别

- `Google One`
  - 更偏面向普通 Google 账号/订阅体系
- `GCP Code Assist`
  - 更偏企业/GCP 相关开发者场景
- `AI Studio OAuth`
  - 更偏 AI Studio 官方 OAuth 场景

#### 账号等级 `tier`

这个字段主要影响：

- 权限等级
- 可用模型/能力范围
- 配额背景

所以它不是装饰字段，而是直接影响账号能力的。

#### 推荐做法

如果你不确定怎么选：

1. 先按你账号真实来源选 OAuth 类型
2. 再按后台提示选匹配的 `tier`
3. 先跑通授权，不要同时叠加太多高级限制

### 7.5.2 Gemini API Key

#### 适合什么场景

适合：

- 你用的是 Google AI Studio 提供的 API Key
- 想直接用 key 方式接 Gemini

#### 主要字段

- `使用帮助`
  - 给你跳转参考说明
- `获取 API Key`
  - 直接去 AI Studio 拿 key
- `Base URL`
  - 默认一般用官方地址
- `API Key`
  - 通常以 `AIza` 开头
- `aistudio tier`
  - 对应 AI Studio 账户等级
- `模型白名单 / 模型映射`
- `自定义模型`
- `池模式`
- `自定义错误码`
- `基础配额`

#### 自定义模型

适合：

- 你要接的模型名不在预设列表里
- 或平台新增模型但后台列表还没更新

不建议乱填，最好确认上游真实支持。

#### 推荐用法

第一次配置建议：

1. 先点帮助确认拿到的是 AI Studio key
2. `Base URL` 保持默认
3. 选正确的 `tier`
4. 先用白名单限制最常用模型

## 7.6 Antigravity

Antigravity 是目前账号管理里最特殊的平台之一，因为它和别的平台最大的不同是：

- 它固定偏向 `模型映射`
- 不推荐按白名单思路理解

当前分为：

- `Antigravity OAuth`
- `Antigravity API Key`

### 7.6.1 Antigravity OAuth

#### 适合什么场景

适合：

- 你走标准 Antigravity OAuth 授权
- 或你手里已经有 `Refresh Token`

#### 支持两条创建路径

第一条：

- 生成授权链接
- 浏览器授权
- 回填授权码
- 系统兑换后创建账号

第二条：

- 手动输入 `Refresh Token`
- 系统逐条校验
- 支持每行一个 RT 批量创建

第二条非常适合：

- 你已经提前拿到了多个 RT
- 要一次性导入多账号

### 7.6.2 Antigravity API Key

#### 适合什么场景

适合：

- 你要接上游中继服务
- 手里有 `Base URL + API Key`

这类方式本质上更像“把 Antigravity 请求转发到一个上游服务”。

#### 主要字段

- `Base URL`
  - 默认可用官方上游地址
  - 也可改成你的中继
- `API Key`
  - 上游服务的密钥

### 7.6.3 Antigravity 的模型映射为什么特别重要

Antigravity 和别的平台不同，它在管理后台里是按“映射优先”来设计的。

也就是说：

- 你更应该思考“请求模型名如何映射到真实模型”
- 而不是单纯去做白名单勾选

典型好处：

- 能兼容很多外部请求模型名
- 统一收敛到少数稳定、可控的实际模型
- 更方便你做成本控制

### 7.6.4 `mixed_scheduling`

这个选项可以理解为：

- 允许 Antigravity 账号参与混合调度

例如：

- 某些统一入口请求会把 Anthropic / Gemini / Antigravity 一起纳入候选池

适合：

- 你就是要把它放进统一调度池里

不适合：

- 你想把 Antigravity 单独隔离使用

### 7.6.5 `allow_overages`

这个选项可以理解为：

- 允许透支 Credits / 超额使用

适合：

- 你明确接受透支策略

不适合：

- 你要严格成本控制

### 7.6.6 默认映射和快捷映射怎么用

推荐做法：

1. 先恢复默认映射
2. 再根据你的实际使用习惯加快捷预设
3. 最后再手工补少数特殊规则

这是最稳的方式。

不要一上来就全手写，否则后期很难维护。

## 9. 推荐的起步配置模板

下面是实际最适合新手的配置思路。

### 9.1 Claude Console 新手模板

- 用 `API Key`
- `Base URL` 保持默认
- 先只开白名单
- 不先开池模式
- 不先加自定义错误码

### 9.2 OpenAI 新手模板

- 优先 `API Key`
- `Base URL` 保持默认
- 先不开自动透传
- 先不开 WS mode
- 白名单只放常用模型

如果你非常确定要保留原始 OpenAI 行为，再考虑开 `自动透传`

### 9.3 Gemini 新手模板

- 如果拿到的是 AI Studio key，就用 `Gemini API Key`
- `tier` 选真实等级
- `Base URL` 默认
- 先只勾少量模型

### 9.4 Bedrock 新手模板

- 优先 `SigV4`
- `Region` 填真实区域
- `Force Global` 先关闭
- 先不加额外高级项

### 9.5 Antigravity 新手模板

- 如果有 RT 或标准 OAuth，就走 `OAuth`
- 如果有中继服务，就走 `API Key`
- 先恢复默认映射
- 先不开 `allow_overages`
- `mixed_scheduling` 只在你明确要混合调度时开启

## 10. 常见误区

### 10.1 误区：先把所有开关都打开

这是最容易把系统配乱的做法。

正确做法：

- 先打通基础认证
- 再逐项加能力

### 10.2 误区：白名单和映射随便选一个都一样

不一样。

- 白名单是“限制”
- 映射是“改写”

### 10.3 误区：Base URL 可以随便改

不能随便改。

一旦改错，可能表现为：

- 认证失败
- 模型不支持
- 响应结构异常

### 10.4 误区：并发数越大越好

不一定。

太大可能会：

- 更快撞上游限流
- 导致错误更频繁

### 10.5 误区：池模式一定更稳

也不一定。

如果你只有一个账号，开池模式没有太大意义。

## 11. 你可以这样理解这套系统

如果把账号管理说得更直白一点：

- `认证方式` 决定“怎么登录上游”
- `Base URL / 代理` 决定“从哪里走”
- `白名单 / 映射` 决定“允许什么模型、最终发什么模型”
- `并发 / 负载 / 优先级` 决定“调度器怎么分流”
- `池模式 / 临时不可调度 / 自定义错误码` 决定“故障时怎么兜底”
- `分组` 决定“这个账号给谁用”

你只要按这个思路看页面，选项就不会显得那么散了。

## 12. 我对你的使用建议

如果你是边用边学，最推荐的方式是：

1. 先只接一种平台，跑通一个账号
2. 再给同平台增加第二个账号，体验调度
3. 再开始加白名单或映射
4. 最后再碰池模式、错误码、临时不可调度

最适合先上手的顺序通常是：

1. `Claude Console API Key`
2. `OpenAI API Key`
3. `Gemini API Key`
4. `AWS Bedrock`
5. `Antigravity`
6. 各类 `OAuth`

原因很简单：

- `API Key` 最直观
- `OAuth` 涉及授权回调、code、refresh token，理解成本更高
