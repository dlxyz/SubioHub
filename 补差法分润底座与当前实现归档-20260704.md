# 补差法分润底座与当前实现归档

## 1. 文档目的

这份文档用于归档当前推广分润模块已经完成的底座建设与后端最小闭环，方便后续继续优化时快速对齐现状、边界和下一步计划。

## 2. 当前目标结构

当前推广链路按三级基础版落地：

1. `agent`
   - 代理
   - 管自己名下渠道分销和用户

2. `distributor`
   - 分销角色
   - 当前已支持登录与后台管理
   - 可作为代理体系下的直接推广层

3. `user`
   - 终端用户
   - 通过代理或分销的邀请码进入系统并发生消费

当前补差法口径：

- 用户直接挂在分销下：
  - 分销拿一级提成
  - 上级代理拿补差提成
- 用户直接挂在代理下：
  - 代理直接拿一级提成
- 平台保留剩余利润

## 3. 数据底座

本轮已经落地的数据库底座共 3 张表：

### 3.1 `promotion_relations`

作用：

- 固定用户当前归属的代理 / 分销关系
- 防止后续变更邀请关系后，历史订单串账
- 为订单完成时的补差法分润生成提供稳定关系快照来源

当前实际用途：

- 注册时若带推广邀请码，会写入归属关系
- 代理创建分销用户时，会自动把该分销绑定到当前代理

### 3.2 `commission_rules`

作用：

- 存放补差法分润规则
- 预留全局规则、代理覆盖规则、分销覆盖规则的扩展能力

当前实际用途：

- 订单完成时优先读取当前生效的全局规则
- 若数据库中不存在全局规则，系统会自动补一条默认全局规则

### 3.3 `commission_split_logs`

作用：

- 存储订单拆分后的分润明细
- 支持一笔订单生成多条收益记录
- 为后续结算、冲正、后台列表、提现联动留底

当前实际用途：

- 订单完成后按补差法自动生成分润底账

## 4. 当前已落地逻辑

## 4.1 用户归属关系写入

已经接入的场景：

1. 用户注册时带推广邀请码
2. 代理后台创建分销用户
3. 绑定邀请人时补写关系表

当前规则：

- 若邀请人是 `agent`
  - 用户直接归属到该代理
  - `direct_parent_role=agent`
  - `binding_source=agent_direct`

- 若邀请人是 `distributor`
  - 用户直接归属到该分销
  - 同时继续向上解析其所属代理
  - `direct_parent_role=distributor`
  - `binding_source=distributor_direct`

- 若关系已锁定
  - 默认不覆盖，避免反复改邀请关系引发历史分润混乱

## 4.2 订单完成后生成补差法分润

当前已接入位置：

- 支付订单完成后的 `markCompleted()` 流程

当前支持订单类型：

- `balance`
- `subscription`

当前执行顺序：

1. 根据消费用户读取 `promotion_relations`
2. 读取当前生效的全局 `commission_rules`
3. 按补差法生成 `commission_split_logs`
4. 保留旧的单级 `commission_logs` 兼容逻辑，不强拆
5. 最后把订单状态标记为 `completed`

说明：

- 当前新逻辑已经能生成新表底账
- 旧表 `commission_logs` 仍保留，避免一次性切断现有功能
- 这属于并行过渡方案，风险更低

## 4.3 默认全局规则

当系统查询不到生效中的全局规则时，会自动创建一条默认规则：

- `name=全局补差默认规则`
- `calc_mode=diff`
- `scope_type=global`
- `agent_target_rate=0.20`
- `distributor_target_rate=0.15`
- `freeze_hours=168`
- `settlement_mode=manual`

当前含义：

- 分销层目标比例：`15%`
- 代理层总目标比例：`20%`
- 若用户挂在分销下，则代理按补差拿 `20%-15%=5%`
- 若用户直接挂在代理下，则代理直接拿 `20%`

## 5. 当前补差法生成规则

### 5.1 用户挂在分销下

示例：

- 订单金额：`100`
- 分销目标比例：`15%`
- 代理目标比例：`20%`

生成结果：

1. 分销收益：
   - `level=1`
   - `beneficiary_role=distributor`
   - `target_rate=0.15`
   - `parent_rate=0`
   - `commission_amount=15`

2. 代理收益：
   - `level=2`
   - `beneficiary_role=agent`
   - `target_rate=0.20`
   - `parent_rate=0.15`
   - `commission_amount=5`

### 5.2 用户直接挂在代理下

示例：

- 订单金额：`100`
- 代理目标比例：`20%`

生成结果：

1. 代理收益：
   - `level=1`
   - `beneficiary_role=agent`
   - `target_rate=0.20`
   - `parent_rate=0`
   - `commission_amount=20`

### 6.7 订单管理、提现管理、推广分润明细的区别

- `订单管理`：面向订单主流程，关注订单本身是否创建、支付、完成、退款，核心对象是 `payment_orders`
- `提现管理`：面向现金流处理，关注已经形成的返佣/分润待结算记录是否需要人工打款或确认结算，核心动作是 `pending -> settled`
- `推广分润明细`：面向分销与规则核对，关注补差法分润是怎么拆出来的、分给了谁、按什么比例生成，核心对象是 `commission_split_logs`

使用边界：

- 订单有问题，先去 `订单管理`
- 要处理佣金打款、人工确认结算，去 `提现管理`
- 要核对某一笔订单为什么这样分润、受益人是谁、比例怎么算，去 `推广分润明细`

当前菜单定位：

- `现金流菜单 -> 订单管理`
  - 看订单支付结果、状态流转、金额和订单侧问题
- `现金流菜单 -> 提现管理`
  - 看待结算记录，并直接执行旧佣金和新分润的手动结算
- `客户管理 -> 推广分润明细`
  - 看新补差法分润底账明细，偏核账、核规则、查明细，不以提现处理为主

## 7. 幂等与稳定性设计

当前已考虑的稳定性点：

1. 关系锁定
   - `promotion_relations.is_locked=true` 时默认不覆盖

2. 分润明细幂等
   - 同一订单、同一收益人、同一层级使用唯一约束
   - 重试时不会重复插入

3. 默认规则兜底
   - 没有全局规则时自动补一条默认规则
   - 避免因为配置缺失导致订单完成链路无法写新分润底账

4. 旧逻辑兼容
   - 暂不删除旧的 `commission_logs`
   - 当前先让新旧逻辑并行，后续再平滑切换

## 10. 后续优化建议

建议后续按这个顺序继续：

1. 先做 `admin` 后台分润明细管理页
   - 直接接 `commission_split_logs`
   - 支持看待结算、已结算、层级收益

2. 再做 `agent` 侧分润明细视图
   - 让代理能看到自己体系内补差法明细

3. 然后做新表结算逻辑
   - `pending -> settled`
   - 支持冻结期、手动结算、批量结算

4. 再做退款冲正
   - 把退款订单对应的新分润底账改为 `reversed / cancelled`

5. 最后再做规则中心
   - 平台全局规则
   - 代理覆盖规则
   - 分销覆盖规则

## 11. 当前结论

当前推广分润已经不再是“只有表结构，没有业务”的状态。

现在已经具备：

- 用户归属关系稳定落库
- 订单完成自动生成补差法分润底账
- 默认全局规则自动兜底
- 后续可直接围绕 `commission_split_logs` 继续做后台页、结算、冲正和规则中心

也就是说，数据库底座和最小后端闭环已经准备好了，下一步继续做功能时，可以直接基于当前实现往上叠加，不需要再回头重折腾底层结构。
